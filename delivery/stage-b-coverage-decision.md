# Stage B ROI 决策 —— 真实覆盖率 sweep（零成本）

> 方法：10 个跨术式真实 case，只跑 FaceMesh 角度分类 + 纯路由（**不渲染、不付费**）。
> 脚本：`backend/scripts/coverage_sweep.py`。模型：`/tmp/focal-p4-asset/face_landmarker.task`。

林真呈(面颊+下巴) · 陈小玲(隆鼻) · 蔡伟玲(下颌线+山根+下巴) · 吕碧英(下巴+苹果肌+太阳穴) ·

黄婧(下巴) · 曾瑜勤(泪沟)

## 核心数据

| 指标 | 值 |
|---|---|
| 抽出部位总数（per-case 去重） | 19 |
| 状态 | covered=19 / degraded=0 / **missing=0** |
| 落 **正面** 板 | 13 (68.4%) |
| 落 **45°** 板 | 6 (31.6%) |
| 落 **侧面** 板 | **0 (0.0%)** |
| **front+45° 覆盖率** | **19/19 = 100%** |
| 照片角度实测分布 | 45°=41 · 正面=24 · no_face=17 · 侧面=5 |

**结论 1：对这 19 个已知部位，侧面板触发 0 次。** front-first 高光路由（你的洞察）+ 现实里正面/斜位照永远有 → 侧面板对常规填充术式低 ROI。

## 但 sweep 暴露了一个更重要的洞 —— 鼻梁类术式抽出 0 部位

- **陈小玲「隆鼻」→ 抽出 0 部位**；蔡伟玲「山根+下巴」→ 只抽到下巴、**漏山根**。
- 根因：atlas 有 `鼻尖`(idx4=鼻头) 且路由 `[正面, 侧面]`，但**抽词只认"鼻尖/鼻基底/鼻翼"，不认"隆鼻/山根/鼻背/鼻梁"**。
- 不能简单 alias 山根→鼻尖：山根/鼻背是**鼻梁(radix/dorsum)**，鼻尖是鼻头，锚点会标错位（违反"不做临时修复"）。

### 这个洞有多大 + 为什么它反过来重新定义 Stage B

- 全库 **8 个**鼻梁类术式（隆鼻/山根/鼻背），约 **17% 患者**，今天**全部产 0 标注**。
- 鼻梁恰恰是**唯一**前后对比"钱景"在侧面的部位 —— 隆鼻效果看的就是侧面鼻背线条。
- 即：通用侧面板 ROI 低（0/19），但**鼻背的侧面标注**是真需求。Stage B 的正当性从"通用侧面剪影"收窄到"鼻背侧面线"。

## 建议

1. **暂缓通用 Stage B 侧面剪影板**（对 16 个非鼻部位 0/19，不值最难那块）。
2. **真正的 Phase-1 缺口 = 建一个正经的 `鼻背` region**（鼻梁 landmark 168→6→197→195 polyline，front 看高光带 + profile 看鼻背线）+ 补 隆鼻/山根/鼻背/鼻梁 抽词。
3. 那个 region 落地后，再决定是否给它配一个**窄范围**侧面能力（只标鼻背线，不做全脸剪影）——这才是 Stage B 真正该长的样子。
